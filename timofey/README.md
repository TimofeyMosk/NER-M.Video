# Журнал работы (timofey) — NER для поиска М.Видео

Документ описывает, что сделано по кейсу «Интеллектуальный поиск М.Видео», какие проводились
проверки, какие были развилки и почему выбран тот или иной путь. В конце — итог «что сделано».

**Задача (уточнённая постановка):** вход модели — **текст поискового запроса** («стиральная
машинка узкая lg»), выход — структурированные факты: `category`, `brand`, `attrs`. Это
NER/query-understanding по запросу, а не по карточке товара. Каталожные `_search`-параметры —
это (1) словарь сущностей и (2) источник weak-разметки, но не выход модели. Приёмка: сервис
отдаёт JSON с фактами за <100мс.

Структура папки:
- `eda/` — разведочный анализ (скрипты `01–04`, отчёты в `eda/output/`, `eda_report.html`).
- этот `README.md` — сквозной журнал пайплайна разметки и обучения.

Код пайплайна разметки лежит в `../labeling/`, пакет обучения — в `../rubert/training/`.

---

## 0. EDA — с чего начинали (папка `eda/`)

Три файла в `cu_ws/`: `query_clicks.parquet` (31.0М строк лога кликов), `skus.pkl` (каталог,
546К офферов, 6963 категории), `sku_desc.parquet` (описания). Ключевые выводы, определившие
дальнейшие решения:

- **Категорию нельзя брать из `sku_subject_id`** — он не равен `category.@id` каталога (совпадает
  лишь ~73% случайно). Категорию резолвим через `sku_id → offer.categories.categoryId → name`.
- **Клики шумные как NER-разметка**: лишь 27% кликов дают буквальное совпадение бренда запроса с
  брендом товара; brand purity кликов ниже category purity.
- **Category purity кликов высокая** — при ≥5 кликах у ~большинства запросов ≥90% кликов на одну
  категорию → категория надёжный weak-label.
- **Нет timestamp/session id** → time-split невозможен, сплит только по query_text (group split).
- **Код-свитчинг и регистр** — частое явление (39.7% запросов кириллица+латиница, 265К запросов
  различаются только регистром).
- **10К+ типов характеристик** — единая плоская схема нереальна, атрибуты индексируем per-category.

Отсюда — план: 4 независимых источника weak-разметки → объединение в BIO-датасет → дообучение
компактного энкодера (rubert-tiny2). Скоуп MVP — **топ-10 категорий по кликам**: Смартфоны,
Стиральные машины, Холодильники, iPhone, Телевизоры, Ноутбуки, Наушники, Пылесосы вертикальные,
Аэрогрили, Планшеты.

**Почему 4 независимых источника, а не один.** Клик, словарь и синтетика ошибаются
по-разному. Собирая их порознь и сверяя на этапе объединения, мы (а) ловим противоречия
(конфликты → на ручной разбор, не в train), (б) не тащим системную ошибку одного источника во
всю разметку.

## 1. Источник A — weak-labels из кликов (`labeling/01_weak_labels_clicks.py`)

**Что делает.** Для каждого уникального `query_text` агрегирует клики: резолвит категорию через
каталог, считает category/brand purity, назначает `category_label` при ≥5 кликов и purity ≥0.9;
brand сохраняет с purity (без порога — фильтрация позже). Результат ограничен топ-10.

**Проверки.**
- Разведка схемы: колонки в parquet обёрнуты в `toValidUTF8(...)`; offer `@id` — строка, `sku_id`
  в кликах — uint64 (джойн через `int(@id)`); 39 битых categoryId → NaN.
- Сверка с контрольными числами EDA.

**Развилка / расследование (важное).** Контроль EDA обещал «category purity ≥0.9 у ~93%
частотных запросов», у нас вышло **80.9%** — расхождение >10pp, промт требовал остановиться и
разобраться. Диагностика (`labeling/diag_category_purity.py`):
- Считая purity по `sku_subject_id` на **всех** 337 654 частотных запросах, получаем 79.5% —
  практически как по имени категории (80.0%). Значит **джойн корректен**.
- Контрольные 93% в EDA были посчитаны на `frequent_q[:20000]` — **первых 20000 запросах по
  алфавиту**; цифры/латиница сортируются раньше кириллицы → это почти целиком конкретные
  модельные запросы («iphone 15 pro 256»), они почти всегда чистые. Воспроизвели ровно 93.0%.
- **Вывод:** контрольное число было нерепрезентативным (смещённая выборка), а не наш джойн.

**Итог источника A:** `weak_labels_clicks.parquet` — 107 879 запросов с category_label.
Контроли сошлись: per-click brand match 27.2% (≈27%), brand purity mean 0.916.

## 2. Источник B — словарный gazetteer (`labeling/02_weak_labels_gazetteer.py`)

**Что делает.** Полностью независимо от кликов (берёт только тексты запросов). Aho-Corasick по
трём справочникам: бренды (`vendor_names.json`), категории (токены имён), атрибуты
(`search_param_names.json`) + маленький транслит-словарь (айфон→iphone и т.п.). Выдаёт спаны с
позициями.

**Развилка — токены брендов vs полные имена.** Контроль EDA (57.2% покрытие по бренду) считался
по **токенам** vendor-имён, а в них много мусора: суб-токены многословных брендов («для» из
«Стиль для Дома», «pro» из «DORN PRO», «смарт» из «Смарт ТВ»). Для weak-label это ложные бренды.
Решение: в датасет матчим **только полные vendor-имена + транслит** (точнее, 49.7% покрытие), а
токен-паритет 57.2% выводим отдельно — он доказывает, что словарь тот же, а не потерян.

**Проверки.** Category coverage 44.8% ≈ контроль 44.9%. Разобрали подозрительные ATTR
(`samsung`/`bosch` как attr) — оказалось не баг: в `search_param_names.json` реально есть параметры
с такими именами. Отмечено, что exact-словарь не ловит морфологию («стиральная машинка» ≠ токен
«стиральные»/«машины») — это ожидаемый потолок и аргумент за ML-этап.

**Итог источника B:** `weak_labels_gazetteer.parquet` — 1 351 783 запроса с ≥1 совпадением.

## 3. Источник C — синтетика из каталога (`labeling/03_synthetic_labels.py`)

**Что делает.** Независимо от кликов/словаря генерирует запросы из реальных комбинаций каталога:
`{category} {brand}`, `{brand} {category}`, `{category} {attr}`, `{brand} {category} {attr}`.
Категории — в натуральных формах (смартфон/телефон, стиралка, тв, ноут, айфон), чтобы звучало как
пользовательские запросы, а не перебор полей.

**Развилки / решения.**
- **Форма пустая во всех топ-10** (это атрибут мебели/матрасов) → используем только Страна и
  нормализованный Цвет (split по `/`).
- **Реальные комбинации, не выдуманные** — бренды/страны/цвета берутся строго из офферов своей
  категории.
- **Объём** подняли до 7 200 (было 3.7К) — в целевом коридоре, заведомо ≤20% train.
- Остаточная грязь Цвета (материалы вроде «алюминий» после split) отмечена как известная.

**Итог источника C:** `synthetic_labels.parquet` — 7 200 примеров, `is_synthetic=true`.

## 4. Объединение в BIO-датасет (`labeling/04_build_dataset.py`)

**Правила мёржа (заданные, не выдуманные).**
- **Категория:** clicks с purity ≥0.9 → берём; иначе полное имя top-10 категории в словаре →
  `source=gazetteer_only`; иначе — без категории.
- **Бренд:** clicks и словарь согласуются → high, берём; расходятся → **conflict=true → golden,
  не в train** (не выбираем один молча); один источник → уверенность по purity (high/medium/low),
  low → в golden.
- **Синтетика** держится отдельным помеченным пулом, её статистика не смешивается с реальной.

**Span alignment → BIO.** gazetteer-спаны берём по готовым позициям; label-level значения
(клики/синтетика) локализуем токен-оконным матчером с Левенштейном (морфология «машинка»~«машина»,
опечатки); бренд из кликов тегируем **только если он реально есть в тексте** (literal_match/
транслит/словарь) — majority-бренд не в запросе позицию не выдумываем; не нашли — сущность
выбрасываем. Сплит — GroupShuffleSplit по query_text.

**Развилка — поймали 2 бага корректности при ревью примеров:**
1. Токен «машины» помечал **посудомоечную** машину как «Стиральные машины». Фикс: для
   gazetteer_only требуем присутствие **всех** токенов имени категории (полное имя).
2. Для gazetteer-only бренда выбор «самого длинного спана» подтягивал мусор-vendor «Смарт ТВ»
   вместо `tcl`. Фикс: приоритет одиночного/латинского токена; + фильтр ATTR от предлогов и
   мета-параметров.

**Итог (первая версия):** train 128 975 (+синтетика), val 21 490, golden 500 (250 conflict +
150 low + 100 rare). conflict 5.0%, выброшено alignment 9.4%. BIO-валидность: 0 нарушений.

## 5. Диагностика датасета (`labeling/05_dataset_diagnostics.py`) — нашли 2 блокера

Прежде чем обучать — замерили качество. **Два серьёзных дефекта:**

1. **🔴 Утечка train↔val 40.2%.** Group split был по сырому `query_text`, но токенизация
   нормализует регистр/пунктуацию → «Холодильник» и «холодильник» разъехались по train/val, а
   токен-текст идентичен. Любой val-F1 был бы завышен ~2x. Плюс 24.4% полных дублей внутри train.
2. **🟠 ATTR — семантически мутный.** Он строился из **имён** параметров, и 11% ATTR-спанов —
   на деле бренды (7.8%) и категорийные слова (3.5%). Класс ATTR учился бы на грязи.

Оба замера воспроизводимы в скрипте (leak по нормализованному токен-тексту, pollution по
пересечению со словарями бренд/категория).

## 6. Правильное извлечение ATTR (`labeling/06_attr_values.py`)

**Развилка — почему ATTR провалился и как чинить.** Корень: матчили **имена** характеристик
(`Цвет`, `Диагональ`), а факт — это **значение** (`белый`, `55`). Замер вскрыл ещё одно:
**единицы почти не пишут** (гб 1.8%, см 0.0%, дюймы 1.5%), хотя цифры у 58.7% запросов — значит
числа **голые**, и их тип задаётся категорией («телевизор 55»→диагональ, «iphone 256»→память).

**Решение — 4 механизма, precision-first:**
1. value-gazetteer из **значений** каталога per-category (цвет, страна, разрешение, тип
   наушников/пылесоса/холодильника, ОС, CPU, энергокласс);
2. лексикон фич-флагов (5g/wifi/oled/qled/nfc/esim/смарт/инверторный/…);
3. regex «число+единица» → тип по единице;
4. голое число — **только если оно ∈ множеству значений каталога** для этой категории (память
   смартфона/планшета, диагональ ТВ/планшета) → так отсекаются модель-коды («iphone 15» → 15∉set).

**Guard'ы** (те же классы ошибок, что чинили в мёрже): значение не тегируется, если это бренд или
категорийное слово («вертикальный»/«стиральная» и морфоформы); голое число — только отдельным
словом. Поймали и починили по ходу: `vacuum_type=вертикальный` (категория), `power=9 w` из
модель-кода `F2V9GW9W`, `diagonal=100` из «100 герц» (добавили `герц` в единицы).

**Итог:** `weak_labels_attr.parquet` — 30 763 запроса (28.5%) с типизированными фактами (память
12.4К, диагональ 10.4К, цвет 4.7К, cpu/network/smart ~2К, …). Каждый спан:
`{text, attr_type, value, start, end, method, confidence}`.

## 7. Пересбор датасета (фикс блокеров) — `labeling/04_build_dataset.py` v2

Правки в 04, чтобы получить датасет, пригодный для честного обучения:
1. **ATTR-источник** переключён с param-имён на типизированный `weak_labels_attr.parquet` (тег в
   BIO обобщённый `B/I-ATTR`; тип восстанавливается детерминированным слоем в проде).
2. **Ключ группировки/дедупа = нормализованный токен-текст** (`' '.join(tokens)`), а не сырой
   query_text → устраняет утечку и внутренние дубли.
3. **Анти-утечка синтетики**: синт.примеры с `norm_key` из val выброшены.

**Проверка после пересбора:** утечка train↔val **0.0%** (было 40.2%), ATTR-категории **0%** (было
3.5%), бренд-в-ATTR 2.9% (было 7.8%; остаток — коллизии значений с именами вендоров, безвредно,
приоритет BRAND>ATTR). Дедуп снял 40 831 дубль. **Итог: train 96 129 (88 994 real + 7 135
synthetic), val 15 705, leak 0.**

## 8. Пакет обучения rubert (`rubert/training/`)

**Развилка — где обучать.** Обучение решено вести **не на этом ноутбуке**, а на внешней
GPU-машине → собран самодостаточный переносимый пакет с данными внутри (сырьё `cu_ws/` на
GPU-боксе не нужно).

Состав: `train_ner.py` (обучение + eval seqeval + замер латентности; «голый» PyTorch без
accelerate/datasets — для максимальной переносимости), `requirements.txt` (torch + transformers
5.14.1 + seqeval), `labels.json` (7 меток BIO), `data/{train,val}.jsonl`, `README.md`.

**Решения по обучению:** модель `cointegrated/rubert-tiny2`; `max_length=64` (p95=8 токенов);
выравнивание меток по сабвордам (первый сабворд = метка, остальные −100); устройство CUDA→MPS→CPU
+ fp16 на CUDA; ~4 эпохи, batch 64, lr 5e-5; лучший чекпойнт по val entity-F1 (seqeval).

**Проверка (смоук-тест на MPS, 500 примеров/1 эпоха):** пайплайн проходит end-to-end (скачивание
модели → обучение → per-type seqeval → сохранение → `metrics.json`). Латентность инференса на CPU
batch=1: **p50 1.6мс, p95 2.2мс** — с огромным запасом под приёмку <100мс. По ходу починили
сериализацию numpy-типов в `metrics.json` и перешли на не-deprecated `torch.amp.GradScaler`.

## 9. Ревью работы коллеги (Pavel) — независимая сверка

Изучили `pavel_nntp/`: у него **query-level классификация** (запрос → доминантная категория/бренд
+ confidence), не token-level NER. Сильные стороны: DuckDB-пайплайн (тянет 31М без OOM), config +
manifest с SHA, стратифицированный сплит, позиционное evidence-взвешивание, `product_family→brand`
маппинг.

**Ключевое расхождение — трактовка `sku_position`.** Pavel считает `position=0` = «без клика» и
выбрасывает 7.39М строк. Мы считаем позицию 0-индексной (каждая строка — клик, 0 = топ выдачи).
**Проверили по данным:** позиция 0 (7.39М) лежит практически **на** кривой затухания рангов 1–6
(степенной фит предсказывает 6.4М, факт 7.39М, отношение **1.15** — не аномальный всплеск, каким
был бы бакет показов), и 9.7% пар (query,sku) встречаются и на позиции 0, и на >0. → Позиция 0 —
это клик по топу, и её отбрасывание теряет сильнейший сигнал и 456К чистых навигационных запросов.
Вывод подан как высокоуверенный, но с рекомендацией сверить семантику с data dictionary М.Видео.
Работы комплементарны: его инженерия + наш token-NER и извлечение атрибутов.

## Итог — что сделано

1. **Полный пайплайн weak-разметки из 3 независимых источников** (клики / словарь / синтетика) →
   объединение по заданным правилам в BIO-датасет с разбором конфликтов (`labeling/01–04`).
2. **Отдельный извлекатель типизированных ATTR-фактов** из значений каталога (`labeling/06`) —
   память/диагональ/габариты/цвет/фичи, с отсечением модель-кодов и загрязнения.
3. **Диагностика и фикс двух блокеров**: устранена утечка train↔val (40.2%→0%) и шум ATTR
   (пересобран на чистый источник). Итоговый датасет: **train 96 129, val 15 705, утечка 0**,
   BIO-валидность 0 нарушений (`labeling/05` + пересбор `labeling/04`).
4. **Переносимый пакет обучения** `rubert/training/` (rubert-tiny2), проверенный смоук-тестом
   end-to-end; латентность инференса на CPU **p50 1.6мс** (≪100мс). Готов к запуску на GPU-машине.
5. **Независимая сверка с работой коллеги** — с расследованием семантики `sku_position` по данным
   (вероятная потеря 7.4М кликов у коллеги) и планом объединения сильных сторон.

**Открытые вопросы / дальше:** (а) сверить семантику `sku_position` с М.Видео; (б) собрать
gold-сет (300–500 примеров) для честной приёмки — текущий val размечен теми же weak-правилами;
(в) запустить полное обучение на GPU и посмотреть per-type F1; (г) при необходимости — типизация
ATTR-подтегов и Go-инференс (отложено).
