## Контекст задачи

Кейс: «Интеллектуальный поиск для e-commerce (М.Видео)». Нужно построить алгоритм/сервис, который принимает поисковый запрос пользователя (короткий, 2-5 слов, часто с опечатками и смесью кириллицы/латиницы, например: "стиральная машинка узкая lg", "холодильники двухкамерные", "Айпад air м3") и менее чем за 100 мс возвращает JSON со структурированными фактами: категория товара, бренд, характеристики (атрибуты).

Данные, с которыми буду работать:
- Лог поиска: 31М строк (запрос, кликнутый товар, позиция в выдаче) — единственный источник реальных пользовательских запросов, но без готовой разметки сущностей.
- Товарный каталог: ~546К товаров, дерево категорий (~7К категорий), у каждого товара бренд + до ~300 характеристик ("Цвет", "Страна", "Материал" и т.д., суммарно ~10К уникальных типов характеристик).
- Тексты описаний товаров (вспомогательно).

Ключевая идея решения: разметку для NER получаем не вручную, а через клики (weak/distant supervision) — если на запрос X чаще всего кликали товары категории Y, считаем запрос X размеченным категорией Y. На этой разметке обучаем модель.

Планируемая архитектура:
1. **Офлайн (Python)**: построение словарей (бренды/категории/атрибуты из каталога) → weak-labeling запросов через клики → дообучение компактного трансформера (кандидат — rubert-tiny2 или похожий маленький русскоязычный энкодер) как token classifier с BIO-тегами (BRAND/CATEGORY/ATTR) → экспорт модели в ONNX.
2. **Онлайн (Go)**: нормализация запроса → быстрый словарный поиск (Aho-Corasick/trie по спискам брендов/категорий/атрибутов, <1мс) → если словарь не всё нашёл — вызов ONNX-модели → сборка JSON-ответа. Бюджет — суммарно <100 мс.

Мой уровень: базовые знания Data Science/NLP, программирую на Python и/или Go, про трансформеры знаю на базовом уровне, но не занимался вплотную ни NER, ни продакшн-инференсом моделей.

## Что нужно в ответе

Дай конспект по следующим темам, **в порядке приоритета** (сначала то, без чего не сдвинуться с места завтра, потом — что желательно знать, но можно почитать позже). На каждую тему — коротко (не более 1 экрана): суть, как это применимо именно в этом кейсе, и если уместно — маленький пример на данных из контекста выше.

**Must-know:**
1. NER как задача token classification: разметка BIO/BIOES, почему именно эта схема, как размечать вложенные/составные сущности (например "стиральная машинка узкая lg" → CATEGORY: "стиральная машинка узкая", BRAND: "lg").
2. Метрики качества NER: entity-level precision/recall/F1 (не путать с token-level accuracy), почему token-level accuracy вводит в заблуждение, что такое seqeval.
3. Weak/distant supervision: что это, типичные проблемы (шумная разметка, false positives/negatives), как фильтровать шум (в нашем случае — по позиции клика и частоте пары запрос-товар).
4. Токенизация субслов (WordPiece/BPE): зачем нужна, как работает на смеси кириллицы/латиницы и опечатках, почему это критично для code-switching запросов.
5. Энкодер-трансформер (BERT-подобная архитектура) для token classification: как классификационная голова работает поверх эмбеддингов токенов, чем маленькие модели (типа rubert-tiny2) отличаются от больших по качеству/скорости.

**Хорошо бы знать:**
6. Aho-Corasick / trie для поиска по словарю: алгоритмическая идея, почему это быстрее наивного substring-поиска, где предел применимости (не ловит опечатки/синонимы).
7. Экспорт модели в ONNX и инференс без Python (квантизация int8, зачем она нужна для латентности на CPU).
8. Проблема паритета токенизатора между Python и другим языком инференса (Go) — почему это частый источник багов в проде.
9. Group split при разбиении на train/test по повторяющимся запросам — почему обычный random split даёт утечку данных.
10. Позиционный bias в кликах (position bias) в поисковых логах — почему клик на 1-й позиции доверия больше, чем клик на 50-й.
