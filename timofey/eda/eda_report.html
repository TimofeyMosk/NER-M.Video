Цель кейса: по поисковому запросу пользователя (2–5 слов, часто с опечатками и латиницей вперемешку с кириллицей) за <100 мс извлечь категорию, бренд и характеристики товара и вернуть их в виде JSON. Вывод: только 27% кликов дают буквальное совпадение «бренд в запросе = бренд кликнутого товара». Это верхняя граница качества для наивного distant supervision по бренду через substring-match — для остальных 73% либо бренда в запросе нет, либо пользователь кликнул не то, что явно спросил (переключился на другой бренд в выдаче). Разметку кликами нужно фильтровать позицией и порогом частоты пары (запрос, sku), а не брать как есть.

Риски и на что обратить внимание в первый день

Только 30.9% sku_id есть одновременно в кликах, каталоге и описаниях
Снапшоты явно сделаны в разное время (часть кликнутых товаров уже не в каталоге / без описания). Нужна стратегия для sku без карточки: fallback на sku_name из query_clicks вместо title/description.
Кликам нельзя доверять как чистой разметке сущностей
Только 27% буквальных brand-match; 40% строк — полные дубликаты (повторные показы/клики). Weak labels из кликов нужно взвешивать по частоте пары (запрос,sku) и позиции, не использовать «как есть».
sku_subject_id ≠ category.@id
Прямое числовое сравнение даёт совпадение лишь у части категорий (похоже на случайные коллизии ID). Категорию нужно получать через sku_id → offer → categories.categoryId, а не по значению subject_id напрямую.
Регистр и код-свитчинг
265К запросов различаются только регистром; 40% запросов содержат латиницу, 40% — смесь кириллицы и латиницы одновременно (бренды/модели). Токенизация и NER-модель должны быть устойчивы к этому без потери сигнала имени собственного (регистр иногда полезен).
10 184 уникальных типа характеристик
Схема характеристик сильно категорийно-специфична — нужно решить на день 1, ограничиваем ли MVP небольшим набором категорий (топ-10 по кликам покрывают значимую долю трафика) или строим category-aware извлечение характеристик.

Рекомендации на завтра

  • Сузить скоуп для MVP: топ-10 категорий по кликам (смартфоны, стиральные машины, холодильники, iPhone, ТВ, ноутбуки, наушники, пылесосы, аэрогрили, планшеты) дают львиную долю трафика и представляют разумное число характеристик на категорию.
  • Построить связующий слой sku_id → category (через offer.categories.categoryId) и sku_id → характеристики (через offer.param), т.к. subject_id из кликов напрямую не резолвится.
  • Слабая разметка для NER: для (query, sku) пар с позицией ≤3 и частотой ≥2 — размечать бренд/категорию/характеристики из товара как псевдо-сущности в тексте запроса (regex/fuzzy match), проверяя на подвыборке вручную.
  • Нормализация текста запроса: lowercase, но с сохранением фичи "оригинальный регистр" отдельно; обработка смешанных латиница/кириллица токенов (транслит-словарь брендов: iphone/айфон, xiaomi/сяоми и т.п.).
  • Baseline: сначала словарный / gazetteer-based NER (список брендов из vendor, категорий из дерева, топ-характеристик per category) + fuzzy-matching — даст быстрый бейзлайн под 100 мс, трансформер — как улучшение точности сверху.
