# Пайплайн обработки данных `query_clicks`

## 1. Назначение документа

Документ фиксирует согласованный пайплайн подготовки данных для задачи интеллектуального поиска и последующего извлечения сущностей из поискового запроса.

Текущий этап включает:

- чтение и проверку исходных данных;
- очистку и нормализацию текста;
- канонизацию брендов и товарных линеек через словари алиасов;
- присоединение товарного каталога;
- обработку запросов без клика;
- схлопывание дублей и агрегацию кликов;
- определение категории, бренда и качества примера;
- подготовку `train` и `valid` без утечки одинаковых запросов;
- сохранение обработанных таблиц, словарей и отчета о качестве.

На этом этапе не выполняются обучение модели, BIO-разметка токенов, подбор архитектуры и экспорт модели.

## 2. Исходные данные

Используем следующие файлы:

| Файл | Назначение |
|---|---|
| `cu_ws/query_clicks.parquet` | Запросы, товары, позиции и связанные с ними поля |
| `cu_ws/skus.pkl` | Основной товарный каталог и связь SKU с категориями |
| `cu_ws/sku_desc.parquet` | Описания и характеристики товаров |
| `timofey/eda/output/category_names.json` | Справочник названий категорий |

Из текущего анализа `query_clicks` известно:

- около 31 млн строк;
- около 1,79 млн уникальных исходных запросов;
- около 333 тыс. SKU;
- около 40% строк являются полными дублями;
- в данных нет надежных полей пользователя, сессии и времени;
- `sku_position = 0` означает, что человек ввел запрос, но не нажал на товар; товара для положительной связи в такой строке нет.

Из-за отсутствия пользователя, сессии и времени нельзя рассчитывать настоящий CTR и нельзя делать временное разбиение.

## 3. Единая схема колонок

Исходные колонки переименовываем в понятный внутренний формат:

```text
query_text
sku_id
sku_name
brand_name
price
sku_subject_id
sku_seo_id
position
```

Типы данных:

| Поле | Тип |
|---|---|
| `query_text` | строка |
| `sku_id` | строка или целое число единого типа во всех таблицах |
| `sku_name` | строка |
| `brand_name` | строка |
| `price` | `float` |
| `position` | целое число |
| идентификаторы категорий | строка или целое число единого типа |

Записи без текста запроса удаляем из рабочего датасета и учитываем в отчете качества. Нулевую цену и пустой бренд автоматически не удаляем: это может быть неполнота каталога.

## 4. Текстовые представления запроса

Для каждого запроса сохраняем три версии:

```text
query_original   — исходный текст без изменений
query_normalized — результат общей очистки
query_canonical  — результат словарной канонизации
```

Пример:

```text
query_original   = "  Айфон-15 Pro  "
query_normalized = "айфон 15 pro"
query_canonical  = "iphone 15 pro"
```

Исходное поле необходимо сохранять для аудита, демонстрации преобразований и возможной разметки позиций символов в будущем.

## 5. Общая нормализация текста

Одинаковые правила применяются к запросам, названиям товаров, брендам, категориям и характеристикам. Исходные значения при этом не перезаписываются.

Порядок обработки:

1. Привести Unicode к единой форме `NFKC`.
2. Заменить неразрывные и другие пробельные символы обычным пробелом.
3. Перевести текст в нижний регистр.
4. Заменить `ё` на `е`.
5. Все виды тире и дефиса заменить пробелом.
6. Удалить кавычки, скобки, emoji и остальные служебные символы.
7. Сохранить буквы всех алфавитов и цифры.
8. Сохранить значимые для моделей символы `+`, `.`, `/`.
9. Несколько последовательных пробелов заменить одним.
10. Удалить пробелы в начале и конце строки.

Примеры:

| Исходный текст | Нормализованный текст |
|---|---|
| `  Айфон-15 Pro  ` | `айфон 15 pro` |
| `Samsung EP-T4511 45W Type-C` | `samsung ep t4511 45w type c` |
| `ХОЛОДИЛЬНИК` | `холодильник` |
| `Стиральная–машина` | `стиральная машина` |
| `USB-C` | `usb c` |
| `S23+` | `s23+` |
| `M.2` | `m.2` |
| `1/2` | `1/2` |

Цифры не удаляем, так как они часто обозначают модель, объем памяти, диагональ, мощность и другие характеристики.

На этом этапе не применяем стемминг и лемматизацию. Они могут изменить названия брендов и моделей и усложнить восстановление исходных сущностей.

## 6. Признаки текста

После нормализации рассчитываем признаки:

```text
query_length_chars
query_length_words
has_digits
has_latin
has_cyrillic
is_mixed_language
has_dictionary_brand
has_dictionary_product_family
```

Эти признаки используются в отчете качества, фильтрации и стратификации.

## 7. Канонизация брендов и товарных линеек

Базовая нормализация не делает `айфон` и `iphone` одинаковыми. Поэтому после нее выполняется словарная канонизация.

Используем небольшой ручной словарь стандартных русских написаний. Бренды и товарные линейки храним отдельно, потому что это разные сущности.

### 7.1. Алиасы брендов

```text
самсунг         -> samsung
эпл             -> apple
сяоми           -> xiaomi
ксиаоми         -> xiaomi
хуавей          -> huawei
хонор           -> honor
реалми          -> realme
леново          -> lenovo
асус            -> asus
эйсер            -> acer
бош             -> bosch
филипс          -> philips
```

### 7.2. Алиасы товарных линеек

```text
айфон           -> iphone
айпад           -> ipad
макбук          -> macbook
аирподс         -> airpods
эпл вотч        -> apple watch
плейстейшен     -> playstation
иксбокс         -> xbox
```

Дополнительно храним связь товарной линейки с брендом:

```text
iphone      -> apple
ipad        -> apple
macbook     -> apple
apple watch -> apple
playstation -> sony
xbox        -> microsoft
```

Таким образом:

```text
"Айфон-15 Pro"      -> "айфон 15 pro"      -> "iphone 15 pro"
"Iphone 15 Pro"     -> "iphone 15 pro"     -> "iphone 15 pro"
"Самсунг S24 Ultra" -> "самсунг s24 ultra" -> "samsung s24 ultra"
```

Правила применения алиасов:

- заменять только целые слова или целые фразы;
- не заменять подстроки внутри других слов;
- сначала проверять самые длинные фразы;
- хранить версию и источник каждого алиаса;
- неизвестные слова оставлять без изменений;
- не применять автоматическую транслитерацию ко всему запросу;
- не использовать fuzzy-мэппинг и автоматическое исправление опечаток;
- добавлять новые значения только вручную после проверки.

В текущей версии исправляются только явно перечисленные стандартные написания. Варианты вроде `айффон`, `айфоун`, `самсуунг` и `ассус` автоматически не заменяются.

Словари храним в JSON и версионируем вместе с кодом. Изменение словаря считается изменением версии датасета, потому что оно влияет на `query_canonical`, агрегацию и разбиение `train/valid`.

## 8. Словари и справочники

В результате подготовки должны быть сформированы следующие словари:

```text
sku_to_name
sku_to_brand
sku_to_category
category_id_to_name
category_id_to_path
category_to_parent
brand_alias_to_brand
product_family_alias_to_family
product_family_to_brand
category_alias_to_category
attribute_alias_to_attribute
attribute_value_alias_to_value
```

Пример:

```text
sku_to_brand[12345] = "samsung"
sku_to_category[12345] = 77
category_id_to_name[77] = "телевизоры"
category_id_to_path[77] = ["электроника", "телевизоры"]
brand_alias_to_brand["самсунг"] = "samsung"
product_family_alias_to_family["айфон"] = "iphone"
product_family_to_brand["iphone"] = "apple"
```

Экспериментально найденные кандидаты не добавляются в эти словари автоматически и не используются при подготовке итогового датасета.

Основным источником бренда и категории является товарный каталог. Поля из лога кликов сохраняются для контроля расхождений, но не имеют приоритета над каталогом.

Категорию связываем через `sku_id` и каталог. Нельзя считать, что `sku_subject_id` автоматически совпадает с идентификатором категории каталога без отдельной проверки.

## 9. Присоединение каталога

Для строк с кликом выполняем присоединение по `sku_id`:

```text
query_clicks.sku_id -> catalog.sku_id
```

После соединения получаем:

```text
sku_id
sku_name_original
sku_name_normalized
brand_id
brand_original
brand_canonical
category_id
category_name
category_path
attributes
```

Контролируем долю SKU, которые не удалось найти в каталоге. Такие строки сохраняем в отдельной таблице ошибок и не используем для построения надежных товарных меток.

## 10. Запросы без клика

Согласованное бизнес-правило:

```text
position = 0 -> пользователь ввел запрос, но не нажал на товар
position > 0 -> есть положительная связь запроса с товаром
```

Для `position = 0`:

- не строим связь `query -> SKU`;
- не используем SKU, бренд или категорию строки как положительную метку;
- если товарные поля заполнены, игнорируем их при создании положительных связей;
- сохраняем запрос как `no_click`;
- считаем `no_click_row_count` на уровне канонического запроса.

Если один запрос встречается и с кликами, и без кликов, храним оба счетчика:

```text
click_row_count
no_click_row_count
```

Отношение этих счетчиков можно использовать как вспомогательный признак, но нельзя называть настоящим CTR из-за отсутствия сессий и идентификаторов событий.

## 11. Обработка дублей

Дубли не удаляем без сохранения их количества. Повторяемость может отражать популярность запроса или связи запроса с товаром.

### 11.1. Полные дубли

Полностью одинаковые строки схлопываем в одну запись и добавляем:

```text
exact_row_count
```

Так мы уменьшаем размер таблицы, но сохраняем исходный вес повторений.

Из-за отсутствия идентификатора события нельзя надежно отличить повторный реальный клик от технического дубля загрузки. Поэтому количество сохраняется и отдельно отражается в отчете.

### 11.2. Повторы пары запрос–товар

Для строк с `position > 0` агрегируем по ключу:

```text
query_canonical + sku_id
```

Рассчитываем:

```text
click_count
min_position
median_position
mean_position
position_score_sum
```

Пример:

```text
"стиральная машина" + SKU 101 встретились 35 раз
```

После агрегации остается одна строка с `click_count = 35`.

## 12. Вес позиции товара

Положительная связь с товаром на высокой позиции обычно сильнее, чем связь с товаром в глубине выдачи.

Для `position > 0` используем предварительную формулу:

```text
position_score = 1 / log2(position + 2)
evidence_score = exact_row_count * position_score
```

Чтобы выбросы не влияли слишком сильно, позицию можно ограничить:

```text
position_capped = min(position, 100)
```

Формула и максимальная позиция должны храниться в конфигурации, чтобы их можно было менять без переписывания пайплайна.

## 13. Агрегация до уровня запроса

Итоговая основная единица датасета — уникальный `query_canonical`.

Для него рассчитываем:

```text
query_canonical
query_normalized_variants
query_original_examples
click_row_count
no_click_row_count
unique_sku_count
top_skus
total_evidence_score
category_distribution
brand_distribution
product_family_matches
text_features
```

Варианты `айфон 15` и `iphone 15` объединяются на каноническом уровне, но их исходные формы сохраняются в списке вариантов.

## 14. Определение категории и бренда

Для каждого запроса суммируем `evidence_score` товаров по категориям и брендам:

```text
category_score = сумма evidence_score товаров категории
brand_score = сумма evidence_score товаров бренда
```

Основные значения:

```text
dominant_category = категория с максимальным category_score
category_confidence = dominant_category_score / total_category_score

dominant_brand = бренд с максимальным brand_score
brand_confidence = dominant_brand_score / total_brand_score
```

Целевой категорией считаем наиболее точную конечную категорию каталога. Дополнительно сохраняем полный путь категории.

Пример:

```text
Запрос: "телевизор samsung 55"

Категории:
телевизоры  -> 0.91
аксессуары  -> 0.06
прочее      -> 0.03

Бренды:
samsung     -> 0.88
lg          -> 0.07
прочие      -> 0.05
```

## 15. Оценка качества примеров

Каждому запросу присваиваем уровень качества:

```text
high_confidence
medium_confidence
low_confidence
no_positive_click
```

Начальные правила:

```text
high_confidence:
  category_confidence >= 0.80
  click_row_count >= 3

medium_confidence:
  category_confidence >= 0.60

low_confidence:
  category_confidence < 0.60

no_positive_click:
  click_row_count = 0
```

Пороговые значения являются конфигурацией и могут быть изменены после анализа распределений.

Для первой обучающей выборки используем `high_confidence` и `medium_confidence`. Неоднозначные и слабые примеры не удаляем: сохраняем их отдельно для анализа и последующих экспериментов.

## 16. Фильтрация

Из основной положительной выборки исключаем:

- пустые запросы после нормализации;
- строки без положительного клика;
- клики по SKU, которых нет в каталоге;
- запросы, состоящие только из служебных символов;
- явно поврежденные или аномально длинные запросы;
- примеры, для которых нельзя надежно определить категорию.

Исключенные записи сохраняются с причиной исключения:

```text
empty_after_normalization
no_positive_click
sku_not_found
ambiguous_category
invalid_schema
outlier_query_length
```

## 18. Разделение на train и valid

Случайно делить исходные строки нельзя. В этом случае одинаковый запрос или его словарный вариант может попасть одновременно в `train` и `valid`, что приведет к утечке данных и завышенной оценке качества.

### 18.1. Единица разделения

Разделение выполняем после агрегации, на уровне:

```text
split_group = query_canonical
```

Это означает, что:

```text
"айфон 15 pro"
"iphone 15 pro"
"Iphone-15 Pro"
```

после очистки и канонизации относятся к одной группе и могут попасть только в одну выборку.

### 18.2. Соотношение

По умолчанию:

```text
train = 80%
valid = 20%
```

Отдельный `test` на текущем этапе не создаем.

### 18.3. Стратификационные признаки

Для каждого `query_canonical` создаем:

```text
dominant_category
frequency_bucket
length_bucket
entity_bucket
quality_bucket
```

Частотность:

```text
rare    = 1-2 положительных строки
medium  = 3-20 положительных строк
popular = больше 20 положительных строк
```

Длина:

```text
short  = 1-2 слова
middle = 3-5 слов
long   = больше 5 слов
```

Словарные сущности:

```text
category_and_brand
category_only
brand_only
product_family
no_dictionary_entities
```

Полный ключ стратификации:

```text
dominant_category
+ frequency_bucket
+ length_bucket
+ entity_bucket
+ quality_bucket
```

Если комбинация слишком редкая для деления, последовательно уменьшаем детализацию:

1. Убираем `quality_bucket`.
2. Убираем `entity_bucket`.
3. Оставляем категорию и частотность.
4. Очень редкие категории временно объединяем в `other_rare` только для стратификации.

Исходная категория при этом всегда сохраняется.

### 18.4. Что проверяет validation

Основной сценарий валидации — новые формулировки запросов при известных категориях и брендах. SKU, категории и бренды могут встречаться в обеих выборках, но один `query_canonical` не может быть одновременно в `train` и `valid`.

Внутри `valid` дополнительно отмечаем `is_hard = true` для сложных примеров:

- смешанная кириллица и латиница;
- опечатки;
- русские алиасы английских брендов;
- редкие категории;
- длинные запросы;
- запросы без точного словарного совпадения;
- запросы с более низкой уверенностью.

Это не отдельное третье разбиение, а диагностическая часть общей validation-выборки.

## 19. Проверки после разделения

Обязательные проверки:

```text
пересечение query_canonical между train и valid = 0
пересечение split_group между train и valid = 0
```

Сравниваем `train` и `valid` по:

- категориям;
- брендам;
- частотности запросов;
- длине запросов;
- доле запросов с цифрами;
- кириллице, латинице и смешанному написанию;
- наличию словарных сущностей;
- уровню качества;
- количеству кликов и уникальных SKU;
- доле запросов без клика в исходной статистике.

Для основных распределений сохраняем абсолютные значения и доли. Существенные отклонения между `train` и `valid` должны блокировать публикацию датасета до выяснения причины.

## 21. Воспроизводимость

Для каждого запуска сохраняем:

```text
версию входных файлов
дату обработки
версию пайплайна
конфигурацию порогов
версию словарей
seed разбиения
число строк на каждом этапе
число исключенных строк по каждой причине
контрольные хеши выходных файлов
```

Seed для разбиения фиксируем в конфигурации. Один и тот же набор входных данных, словарей и параметров должен приводить к одинаковым `train` и `valid`.

## 22. Итоговая последовательность

```text
исходные файлы
-> проверка схемы и типов
-> сохранение исходных текстовых полей
-> общая нормализация текста
-> канонизация брендов и товарных линеек
-> присоединение товарного каталога
-> разделение строк с кликом и без клика
-> схлопывание полных дублей с сохранением веса
-> агрегация query_canonical + sku_id
-> расчет веса позиции
-> агрегация до query_canonical
-> определение категории и бренда
-> оценка уверенности и качества
-> фильтрация с сохранением причин
-> групповая стратификация 80/20
-> проверки отсутствия утечек
-> сохранение Parquet, JSON, YAML и HTML
```

## 23. Зафиксированные решения

1. `sku_position = 0` — запрос без клика, а не первая позиция товара.
2. Положительные связи `query -> SKU` строятся только для `position > 0`.
3. Все виды тире и дефиса заменяются пробелом.
4. Все текстовые поля приводятся к нижнему регистру.
5. `ё` заменяется на `е`.
6. Цифры и символы `+`, `.`, `/` сохраняются.
7. `айфон` и `iphone` канонизируются в `iphone`.
8. Бренды и товарные линейки хранятся в разных ручных словарях.
9. Fuzzy-мэппинг, автоматическая транслитерация и автоматическое исправление опечаток не используются.
10. Экспериментальные кандидаты алиасов не входят в итоговый словарь.
11. Полные дубли схлопываются, но их количество сохраняется как вес.
12. Основная единица итогового датасета — `query_canonical`.
13. Разделение выполняется как `80% train / 20% valid` по группам `query_canonical`.
14. Один канонический запрос не может попасть в обе выборки.
15. На текущем этапе модель не обучается.
