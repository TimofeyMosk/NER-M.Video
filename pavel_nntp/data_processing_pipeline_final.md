# Итоговый пайплайн обработки `query_clicks`

## 1. Загрузка данных

Используем:

- `query_clicks.parquet` — запросы и клики;
- `skus.pkl` — товары, бренды и категории;
- `sku_desc.parquet` — описания и характеристики товаров.

Проверяем схему, типы данных, пустые значения и количество строк.

## 2. Очистка текста

Для запросов, товаров, брендов и категорий:

1. Сохраняем исходный текст в поле `*_original`.
2. Приводим Unicode к форме `NFKC`.
3. Переводим текст в нижний регистр.
4. Заменяем `ё` на `е`.
5. Все тире и дефисы заменяем пробелом.
6. Удаляем кавычки, скобки и служебные символы.
7. Сохраняем буквы, цифры и символы `+`, `.`, `/`.
8. Несколько пробелов заменяем одним.
9. Удаляем пробелы по краям.

Пример:

```text
"  Айфон-15 Pro  " -> "айфон 15 pro"
```

## 3. Словарь стандартных названий

После очистки применяем небольшой ручной словарь стандартных русских написаний:

```text
айфон       -> iphone
айпад       -> ipad
макбук      -> macbook
аирподс     -> airpods
плейстейшен -> playstation

самсунг     -> samsung
эпл         -> apple
сяоми       -> xiaomi
ксиаоми     -> xiaomi
хуавей      -> huawei
хонор       -> honor
асус        -> asus
```

Замена выполняется только для целого слова или фразы. Автоматический поиск опечаток и fuzzy-мэппинг не используем.

Храним три версии запроса:

```text
query_original
query_normalized
query_canonical
```

## 4. Разделение кликов

```text
sku_position = 0 -> запрос без клика
sku_position > 0 -> пользователь нажал на товар
```

Строки без клика сохраняем отдельно и не используем для связи запроса с SKU, брендом или категорией.

## 5. Присоединение каталога

Для строк с кликом присоединяем каталог по `sku_id` и получаем:

```text
sku_name
brand
category_id
category_name
category_path
attributes
```

Основным источником бренда и категории считаем каталог.

## 6. Обработка дублей

Полные дубли схлопываем в одну строку, но сохраняем их количество:

```text
exact_row_count
```

Затем агрегируем по паре:

```text
query_canonical + sku_id
```

Считаем:

```text
click_count
min_position
median_position
mean_position
```

Таким образом, повторы не раздувают таблицу, но их частота не теряется.

## 7. Агрегация до запроса

Для каждого `query_canonical` рассчитываем:

```text
click_count
no_click_count
unique_sku_count
top_skus
распределение брендов
распределение категорий
основной бренд
основную категорию
уверенность определения
```

Целевой категорией считаем конечную категорию каталога, полный путь категории сохраняем отдельно.

## 8. Фильтрация

В основную выборку не включаем:

- пустые запросы;
- запросы без положительного клика;
- SKU, которых нет в каталоге;
- явно поврежденные запросы;
- запросы, для которых категория определяется неоднозначно.

Исключенные строки сохраняем отдельно с причиной исключения.

## 9. Разделение на train и valid

Используем:

```text
train = 80%
valid = 20%
split_group = query_canonical
```

Один и тот же канонический запрос может находиться только в одной выборке.

Стратификацию выполняем по:

- основной категории;
- частотности запроса;
- длине запроса.

Редкие категории временно объединяем в `other_rare` только для стратификации. Исходную категорию не изменяем.

После разделения проверяем:

```text
пересечение query_canonical между train и valid = 0
```

## 10. Результат

Все результаты сохраняем в `pavel_nntp/data/`:

```text
query_dataset.parquet
no_click_queries.parquet
rejected_rows.parquet
train.parquet
valid.parquet
aliases.json
processing_config.yaml
data_processing_report.html
```

Итоговая последовательность:

```text
загрузка
-> очистка текста
-> словарь стандартных названий
-> разделение кликов и no-click
-> присоединение каталога
-> схлопывание дублей
-> агрегация query + SKU
-> агрегация до query_canonical
-> фильтрация
-> стратифицированное разбиение 80/20
-> сохранение данных и отчета
```
