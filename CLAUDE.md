# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Case overview

Кейс буткемпа: «Интеллектуальный поиск для e-commerce (М.Видео)» — NLP/NER по поисковым запросам.

Поисковая система М.Видео должна определять категорию, бренды и характеристики товара, чтобы точно отвечать на запросы пользователей. Точность выдачи напрямую влияет на конверсию в покупку. Команде предстоит создать алгоритм, который правильно извлекает сущности из поисковых запросов.

**Важное уточнение постановки (чтобы не путать с соседней задачей):** вход модели — это **текст поискового запроса** (например, "стиральная машинка узкая lg"), а не карточка товара. Задача — NER/query understanding по запросу: category/brand/attrs, которые пользователь имел в виду. Извлечение характеристик из карточек товара (`_search`-параметры в `skus.pkl`) — это уже готовые данные, они используются только как (1) словарь сущностей и (2) источник weak-разметки запросов через клики, но сами по себе не являются выходом нашей модели. Построение самого поискового движка (индекс, ранжирование выдачи) — вне скоупа кейса; наш результат — JSON с фактами, который в проде подавался бы на вход такому движку.

**Успешный результат**: словари, модели или сервис на Python или Go, который получает «запрос» и менее чем за **100 мс** выдаёт структурированный ответ с фактами в формате JSON.

## Repo layout

Team bootcamp repo, organized by person/component rather than a single app tree:

- `cu_ws/` — raw data, **gitignored**, present only locally (see Data below). Never `git add` anything here.
- `timofey/eda/` — EDA scripts, numbered `01`–`04` (run in that order), plus their captured stdout under `eda/output/`. Also holds derived gazetteer JSONs (`vendor_names.json`, `category_names.json`, `search_param_names.json`) and `eda_report.html` (rendered writeup with graphs).
- `nikita/`, `pavel/` — other team members' working directories, currently empty.
- `inference/` — Go module for the online inference service (currently a skeleton, see Architecture below).

No shared dependency manifest exists (no `requirements.txt`/`pyproject.toml`/lockfile). EDA scripts need `pandas`, `numpy`, `pyarrow` (for parquet) — check they're installed in whatever environment you use before running.

## Commands

**Run an EDA script** (from repo root, so the relative `cu_ws/...` paths resolve):
```
python3 timofey/eda/04_mvp_signals.py
```
Existing scripts write their stdout into a matching file under `timofey/eda/output/` (e.g. `04_mvp_signals.py` → `output/04_mvp_signals.txt`) — follow that convention for new numbered scripts:
```
python3 timofey/eda/05_foo.py > timofey/eda/output/05_foo.txt
```

**Go inference service** (module `inference`, Go 1.26):
```
cd inference && go build ./...
cd inference && go run ./cmd
```
No tests exist yet in `inference/`.

## Data (`cu_ws/`) — EDA от 18.07.2026

Три файла, снятые не совсем одномоментно (см. пересечения ниже). Полный отчёт с графиками собирается из скриптов `timofey/eda/0{1,2,3,4}_*.py` → `timofey/eda/eda_report.html`.

### query_clicks.parquet — 31.0М строк, 4.4 GB
Лог поиска: строка = (запрос, кликнутый sku, позиция в выдаче на момент клика).
Колонки: `sku_id, sku_name, sku_brand_name, sku_price, sku_subject_id, sku_seo_id, query_text, sku_position`.
- 1.79М уникальных query_text, 332.8К уникальных sku_id, 7 150 брендов.
- 40% строк — полные дубликаты (повторные показы/клики одной и той же пары).
- Медиана длины запроса — 3 слова (14–34 символа) — короткие телеграфные запросы.
- 58.7% запросов содержат цифры, 39.7% — смешение кириллицы и латиницы одновременно, 23.5% — только латиница.
- 265 014 запросов различаются только регистром (нужна lowercase-нормализация с сохранением сигнала регистра отдельно).
- Только **27%** кликов дают буквальное совпадение бренда в запросе с брендом кликнутого товара → нельзя брать клик как чистую NER-разметку без фильтрации по позиции/частоте пары.
- 28.4% строк — клик по товару без бренда (общие категорийные запросы).
- `sku_position`: сильный positional bias (0 → 23.8% кликов, дальше быстрый спад, но хвост тянется до 5893).
- **Важно**: `sku_subject_id` НЕ равен `category.@id` из skus.pkl напрямую (совпадает лишь ~73% случайно, а среди топ-20 по объёму — только 1). Категорию нужно резолвить через join `sku_id → offer.categories.categoryId → category name`.
- Топ-10 категорий по кликам (после резолва): Смартфоны, Стиральные машины, Холодильники, iPhone, Телевизоры, Ноутбуки, Наушники, Пылесосы вертикальные, Аэрогрили, Планшеты — разумный скоуп для MVP.
- В логе нет ни timestamp, ни session/user id — только `(sku_*, query_text, sku_position)`. Следствие: **time-based split невозможен**, сплит train/test делать по query_text (group split), иначе утечка (один и тот же запрос попадёт и в train, и в test через разные строки).

### skus.pkl — товарный каталог (YML/Yandex Market формат)
Структура: `dict['yml_catalog']['shop']` → `categories.category` (список) и `offers.offer` (список).
- 6 963 категории (96 корневых), 546 069 офферов, 6 765 уникальных vendor.
- Оффер: `@id, @available, vendor, model, name, categories.categoryId, param[]`.
- `param` — список `{'@name', '#text', '@unit'?}`; параметры с суффиксом `_search` — это индексируемые характеристики ("факты").
- 12 999 уникальных сырых имён param, из них 10 184 с суффиксом `_search`.
- Медиана параметров на оффер — 27 (p95=100, max=301) — сильно разное число характеристик по категориям.
- Топ-категории **по числу SKU** — не электроника, а текстиль/товары для дома (матрасы, чехлы для смартфонов, фотошторы) — не путать с топ-категориями по кликам (см. выше), это разные срезы.
- **Значения атрибутов неоднородны по чистоте.** `Форма` (85 значений) и `Страна` (92) — чистые закрытые словари, готовы для классификации. `Цвет` (1558 значений) и `Материал` (675) — грязные, много составных строк вида `"красный/ нержавеющая сталь"` — нужен парсинг/сплит по `/` перед использованием как меток.
- Практический вывод: справочник характеристик нужно индексировать per-category, единая плоская схема на 10К+ типов сущностей нереалистична для MVP.

### sku_desc.parquet — 1.177М строк
Колонки: `sku_id, title, description` (description — HTML-текст, медиана 630 символов). Без пропусков и дублей по sku_id.

### Пересечение по sku_id между тремя файлами
- clicks ∩ catalog (skus.pkl): 72.6%
- clicks ∩ sku_desc: 47.5%
- во всех трёх одновременно: 30.9% (102 945 sku)
→ нужен fallback на `sku_name` из query_clicks, когда для sku нет карточки/описания.

### Weak-labeling качество (из `timofey/eda/04_mvp_signals.py` / `output/04_mvp_signals.txt`)
- Gazetteer coverage (простое совпадение токена запроса со словарём, без fuzzy): по вендорам — 57.2% запросов, по токенам названий категорий — 44.9%. Это реалистичный потолок для чисто словарного baseline без ML — прилично, но не покрывает опечатки/незнакомые бренды.
- **Category purity кликов — очень высокая**: для запросов с ≥5 кликами у 93% ≥90% кликов приходится на одну категорию (медиана — 100%). Значит **категория — надёжный weak-label** прямо из кликов.
- **Brand purity кликов — заметно ниже**: медиана 85.7%, среднее 82.5% — бренд как weak-label шумнее категории, но всё ещё полезен (используем с весом/порогом).

### Открытые риски перед стартом
1. Клики как weak-labels для NER нужно взвешивать (позиция, частота пары query+sku), не использовать напрямую.
2. subject_id из кликов не мапится на category id каталога напрямую — только через sku_id.
3. Код-свитчинг (кириллица/латиница вперемешку) и регистр — частое явление, учитывать в токенизации.
4. 10К+ типов характеристик в каталоге — для сервиса с ответом <100мс нужен явно ограниченный per-category набор атрибутов, а не полный список.

## Planned architecture

### План MVP (в рамках ограничения <100мс, JSON с фактами)
1. **Скоуп**: топ-10 категорий по кликам (Смартфоны, Стиральные машины, Холодильники, iPhone, Телевизоры, Ноутбуки, Наушники, Пылесосы вертикальные, Аэрогрили, Планшеты) — покрывают львиную долю трафика, разумное число атрибутов на категорию.
2. **Разметка**: category — weak label с высокой уверенностью (majority sku_subject_id→category per query, порог purity ≥0.9); brand — weak label с меньшей уверенностью, использовать вместе с literal substring match как доп. сигналом; характеристики — сначала только «чистые» атрибуты (Страна, Форма, нормализованный Цвет после сплита по `/`), остальное — stretch goal. Сплит train/val/test — по уникальным query_text (GroupShuffleSplit), не по строкам.
3. **Baseline (день 1)**: Aho-Corasick / trie по словарям бренд+категория+чистые атрибуты (из `vendor_names.json`, `category_names.json`, `search_param_names.json`) + нормализация (lowercase, транслит-словарь брендов iphone/айфон, xiaomi/сяоми и т.п., сплит составных значений атрибутов). Даёт объяснимый, супербыстрый (<1мс) фундамент с покрытием ~50-60% по брендам/категориям.
4. **ML-этап**: дообучить компактный трансформер (кандидат — rubert-tiny2 или аналогичный маленький русскоязычный энкодер) как token classifier (BIO-теги: BRAND/CATEGORY/ATTR) на weak-labels + синтетических запросах, сгенерированных шаблонами из атрибутов каталога (для покрытия редких/отсутствующих в логах комбинаций). Модель нужна, чтобы закрыть то, что не поймал словарь: опечатки, code-switching, незнакомые бренды.
5. **Оценка**: entity-level F1 на held-out по query_text; отдельно замер задержки p50/p95 на целевом железе — это часть приёмки (<100мс), а не только качество.

### Python-обучение + Go-инференс — технически стандартный паттерн
- Обучение/дообучение — Python (HuggingFace transformers/PyTorch), это даёт доступ ко всей экосистеме NLP-тулинга.
- Экспорт модели в **ONNX** (или TorchScript) после обучения.
- Инференс в Go — через биндинги к ONNX Runtime (Go-обёртки над C API onnxruntime) — на CPU с квантованной (int8) маленькой моделью инференс укладывается в единицы-десятки мс, что оставляет запас под 100мс вместе с токенизацией и сборкой JSON.
- **Главный технический риск** — паритет токенизатора: Python (HF `tokenizers`, Rust-based) и Go должны токенизировать идентично. Варианты: (а) Go-биндинги к тому же Rust-крейту `tokenizers` через cgo, (б) переисполнение WordPiece/BPE на Go по словарю модели (спецификация простая, но легко разойтись в edge cases — нужно тестировать на паритет с Python построчно). Это стоит проверить на игрушечном примере в первый день, чтобы не упереться в это на финише.
- Gazetteer-часть (Aho-Corasick по словарям) реализуется в Go нативно и без Python-зависимости — имеет смысл держать её как отдельный быстрый первый проход перед вызовом модели.
- Итоговый сервис: Go принимает запрос → gazetteer fast-path → (если нужно) ONNX-инференс → сборка JSON с фактами. Полностью реализуемо в заявленном стеке (Python/Go), паттерн распространён в проде.

### `inference/` Go module — текущее состояние: только скелет
- `cmd/main.go` — entrypoint, пока только `package main` (без логики).
- `internal/domain`, `internal/app`, `internal/ports`, `internal/adapters` — пустые директории, задел под hexagonal/ports-and-adapters layout для пайплайна выше. При добавлении кода следовать этому делению: `domain` — базовые типы (query, entities, facts), `ports` — интерфейсы, от которых зависит app (matcher, model runner), `adapters` — конкретные реализации (Aho-Corasick gazetteer, ONNX runtime client), `app` — оркестрация use-case'ов.
